# /new-robot-profile — Wire a real robot into ExampleRobotProfile

Use this playbook when a team wants to run their actual robot's hardware (Falcons, Krakens, Pigeon 2, NavX) instead of the simulation defaults.

---

## Step 0 — Understand what already exists

`src/main/java/frc/robot/example/ExampleRobotProfile.java` is the team's entry point. It already:
- Defines a `SwerveConstants.Builder` with placeholder CAN IDs and measurements
- Branches on `RobotBase.isReal()` in `createDrive()`
- Returns `SwerveFactory.build(CONSTANTS, BLUE_START)` in sim (IronMaple physics)
- Throws `UnsupportedOperationException` on hardware until the team wires real IO

The team only needs to fill in **actual values** and **unwrap the hardware IO block**.

---

## Step 1 — Fill in SwerveConstants

Replace the placeholder values in `CONSTANTS` with the real robot's measurements.

```java
private static final SwerveConstants CONSTANTS = new SwerveConstants.Builder()
    .moduleType(ModuleType.TALON_FX)   // Falcon 500 / Kraken X60; or SPARK_TALON for NEO drive + Falcon steer
    .gyroType(GyroType.PIGEON2)        // or NAVX
    .gyroCanId(1)                      // Pigeon 2 CAN ID

    // Physical geometry — measure the actual robot
    .trackWidth(Inches.of(22.75))      // left-to-right wheel centre distance
    .wheelBase(Inches.of(22.75))       // front-to-back wheel centre distance
    .wheelRadius(Inches.of(2.0))       // actual wheel radius (not nominal)

    // Speed limits — check TunerX or motor characterisation
    .maxLinearSpeed(MetersPerSecond.of(4.5))
    .maxAngularSpeed(RadiansPerSecond.of(2 * Math.PI))

    // Weight and bumper size — used for IronMaple physics in simulation
    .robotMass(Pounds.of(125))         // total robot weight including battery and bumpers
    .bumperLength(Inches.of(30))       // outside-to-outside front-to-back
    .bumperWidth(Inches.of(30))        // outside-to-outside left-to-right

    // CAN IDs: [driveId, steerId, encoderId] per module
    .frontLeftIds(1, 2, 3)
    .frontRightIds(4, 5, 6)
    .backLeftIds(7, 8, 9)
    .backRightIds(10, 11, 12)

    // CAN bus (leave "" for RIO bus; use CANivore bus name for Phoenix Pro)
    .canBusName("")
    .odometryFrequency(Hertz.of(100))  // 250 Hz for CANivore, 100 Hz for RIO bus
    .build();
```

**Required static imports at the top of `ExampleRobotProfile.java`:**
```java
import static edu.wpi.first.units.Units.Hertz;
import static edu.wpi.first.units.Units.Inches;
import static edu.wpi.first.units.Units.Pounds;
import static edu.wpi.first.units.Units.MetersPerSecond;
import static edu.wpi.first.units.Units.RadiansPerSecond;
```

## Step 2 — Set the starting pose

```java
private static final Pose2d BLUE_START = new Pose2d(1.5, 5.5, new Rotation2d());
```

Choose the Blue-alliance starting position for this year's game. The Red-alliance pose is derived automatically by `SwerveRobotContainer.getAllianceStartPose()` (mirrors x across the field centre line and adds 180° to the heading).

## Step 3 — Wire hardware IO (TALON_FX path)

`SwerveFactory.build()` throws for `TALON_FX` in REAL mode. The team must provide their CTRE TunerX `SwerveModuleConstants` and use the real IO classes directly.

Replace the `throw` in `createDrive()` with:

```java
if (RobotBase.isReal()) {
    GyroIO gyro = new GyroIOPigeon2(CONSTANTS);
    ModuleIO[] modules = {
        new ModuleIOTalonFXReal(CONSTANTS, TunerConstants.FrontLeft),
        new ModuleIOTalonFXReal(CONSTANTS, TunerConstants.FrontRight),
        new ModuleIOTalonFXReal(CONSTANTS, TunerConstants.BackLeft),
        new ModuleIOTalonFXReal(CONSTANTS, TunerConstants.BackRight),
    };
    return new AkitSwerveDrive(CONSTANTS, gyro, modules);
}
```

**Required imports:**
```java
import org.frc5010.common.drive.swerve.akit.GyroIO;
import org.frc5010.common.drive.swerve.akit.GyroIOPigeon2;
import org.frc5010.common.drive.swerve.akit.ModuleIO;
import org.frc5010.common.drive.swerve.akit.ModuleIOTalonFXReal;
// plus the team's own TunerConstants class from CTRE TunerX output
```

`TunerConstants` is generated by Phoenix Tuner X's Swerve Project Generator. Each `TunerConstants.FrontLeft` etc. is a `SwerveModuleConstants` object that carries gear ratios, motor IDs, gains, and encoder offsets.

## Step 3 (alternate) — Wire hardware IO (SPARK_TALON path)

For a NEO drive (SparkMax) + Falcon/Kraken steer (TalonFX) robot, use `.moduleType(ModuleType.SPARK_TALON)` and:

```java
if (RobotBase.isReal()) {
    GyroIO gyro = new GyroIONavX();   // or GyroIOPigeon2(CONSTANTS)
    ModuleIO[] modules = {
        new ModuleIOSparkTalon(CONSTANTS, 0 /* module index */),
        new ModuleIOSparkTalon(CONSTANTS, 1),
        new ModuleIOSparkTalon(CONSTANTS, 2),
        new ModuleIOSparkTalon(CONSTANTS, 3),
    };
    return new AkitSwerveDrive(CONSTANTS, gyro, modules);
}
```

Module index 0–3 maps to FL, FR, BL, BR (matches the CAN ID arrays in `SwerveConstants`).

## Step 4 — Test the sim path first

Before deploying to hardware, verify the sim path works with real robot parameters:

```powershell
# Default — uses ExampleRobotProfile, IronMaple physics, real robot mass/geometry
.\gradlew.bat simulateJava

# Or the visual auto-test sequence (drives a box pattern)
.\gradlew.bat simulateJava -PvisualTest
```

If the robot in Glass looks physically plausible (correct size, realistic acceleration), the constants are good.

---

## Step 5 — Deploy and test on hardware

```powershell
.\gradlew.bat deploy
```

Checklist:
- [ ] All module CAN IDs match the physical robot
- [ ] Encoder offsets are set via TunerX (for TalonFX) or burned to SparkMax flash (for SPARK_TALON)
- [ ] Gyro powers on and returns a stable yaw reading
- [ ] Each module drives forward when commanded with `ChassisSpeeds(1, 0, 0)` (check sign)
- [ ] Robot drives straight forward — no drifting sideways

---

## Common mistakes

| Mistake | Symptom | Fix |
|---------|---------|-----|
| Wrong `trackWidth` / `wheelBase` | Robot curves instead of driving straight | Measure wheel centres, not frame edges |
| Wrong `wheelRadius` | Odometry distance off by a fixed ratio | Use the worn-down actual radius, not the nominal spec |
| Modules wired out of order (FL/FR/BL/BR) | Robot spins when asked to go straight | Swap module index or CAN IDs to match physical layout |
| `canBusName` wrong | All TalonFX signals timeout | Match the CANivore device name in Phoenix Tuner |
| `odometryFrequency` too high for RIO bus | Odometry jitter, CAN utilisation >90% | Use 100 Hz for RIO bus, 250 Hz only for CANivore |
